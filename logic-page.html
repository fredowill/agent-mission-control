The Trust Problem

Mission Control is useless if you can't trust it. Right now, when you look at an "active" card, you don't know if that agent is actually running. When you click into its prompts, you don't know if those are the real prompts. That has to end. Every fix below serves one goal:

1
If a card says "active", that session is actually running right now. Not 26 minutes ago. Not a ghost from another project. Right now, in a terminal tab on this machine, doing work.
2
If you click on a card and see prompts, those are YOUR prompts for THAT session. Not duplicated from another session. Not missing half the conversation. Not flooded with system junk. Your words, in order, complete.
3
When you close a terminal tab, that card moves to done. Within 2 minutes, not 6 hours. No zombie sessions. No manual cleanup. It just works.

Open — Performance & Reliability
Remaining Bugs
Fixed /resume in New Terminal Gets Stuck as Done

Was: When closing a terminal and doing /resume in a new one, the card stayed "Done" forever. The hook only resolved PID when null — a dead stored PID was never re-checked.

Now: Both hook.js and prompt-hook.js run a quick tasklist liveness check (~50ms) on the stored PID. If dead, it's nulled and re-resolved via PowerShell process-tree walk. A resumeCount field tracks how many times a session was resumed in a new terminal, shown as a blue badge on the card.

Major 56+ Synchronous Disk Reads Every 2 Seconds

readAgents() is called on the dashboard polling interval. Every call does synchronous readFileSync on every state file, every log file, every prompt file, and missions.json once per agent.

Partially mitigated: getLiveClaudePids() now has a 3-second cache, so the expensive tasklist call isn't repeated every poll. However, readAgents() itself still does sync reads of all state files on every cycle.

Remaining fix: Cache readAgents() results in memory with a 3-5 second TTL. Invalidate on write (when a hook fires and updates a state file via the API). Also cache missions.json in memory instead of re-reading per agent.
Minor — Cleanup
Quality-of-Life Issues
Minor No Auto-Archive for Stale Sessions

Sessions older than 24h get the archived flag but are never cleaned up. They stay in the state directory and continue to be read on every poll cycle. Over time this means dozens of dead sessions accumulating.

Fix: Add a cleanup routine that runs once per hour. Sessions marked archived (>24h, done) should have their state files moved to an archive/ directory (or just deleted, since logs and prompts are the real historical record).
Minor state:'done' Never Written by Current Hooks

The Stop hook writes state: 'idle' (or 'thinking'), never 'done'. The 'done' state is only assigned by readAgents() in memory based on the process-count heuristic. This means the state file on disk never says "done" — it's always a derived property.

This creates inconsistency: old sessions that were manually set to done (if any) have a different data shape than sessions marked done by the heuristic.

Fix: With timestamp-based liveness, "done" becomes a derived state in the server (last event > threshold). This is fine — just make sure the API response clearly distinguishes between the on-disk state and the derived liveness state. Don't pretend the file says "done" when it doesn't.

How It Works (Verified)

This documents the actual Windows Terminal + Claude Code process model as verified through testing. The original analysis had errors (see Corrections tab) — this version reflects what we confirmed empirically and shipped.

How Claude Code Runs on Windows

Process Lifecycle

When you open a terminal tab and type claude, here's what happens:

  • Windows Terminal (or VS Code terminal) spawns a shell process (bash.exe, powershell.exe, etc.)
  • The shell spawns claude.exe as a child process
  • claude.exe generates a UUID session ID at startup — this is the only stable identifier for this conversation
  • Every time the agent makes a tool call, Claude Code invokes hooks by spawning a new child process (e.g., node hook.js) and piping JSON to its stdin
  • Each hook invocation is a separate node.exe process with its own PID. The hook's process.ppid is NOT the claude.exe PID — it's an intermediate process

What Happens When You Close a Tab

This is the critical gap that breaks everything:

  • Windows Terminal sends CTRL_CLOSE_EVENT to the console process group
  • The shell receives the signal and terminates
  • claude.exe is killed as part of the process tree — there is no graceful shutdown
  • Node.js SIGTERM/SIGINT handlers do not reliably fire on Windows when the parent terminal is killed
  • The Stop hook never fires because there's no agent turn to end
  • Result: Mission Control has absolutely no signal that the session ended. The state file just sits there forever with its last known state.

The Shipped Architecture (PID-Based)

The timestamp-only approach proposed here originally was wrong — it failed during /compact, long thinking, and idle sessions where the terminal is still open. See Corrections tab for details. Here is what actually shipped:

// HOOK (once per session): walk process tree to find claude.exe
if (!state.claudePid) {
  claudePid = findClaudePid(process.ppid); // PowerShell, ~1.2s
}

// SERVER (every poll): one tasklist call for ALL sessions
const livePids = getLiveClaudePids(); // ~130ms, cached 3s

// Per session: PID alive = active. PID dead = done.
if (a.claudePid && livePids.has(a.claudePid)) {
  a.active = true;  // terminal window is open
} else {
  a.done = true;    // terminal was closed (or no PID yet)
}

// Sessions sharing a claudePid are MERGED into one card
// (/plan → implement, /compact = same terminal = same card)

This works because:

  • claude.exe PID is stable for the lifetime of a terminal tab
  • When the tab closes, the process dies — next poll detects it
  • Works during /compact, idle, long responses — PID stays alive
  • Session ID changes (/plan, /resume) don't create new cards — same PID = same card
  • Zero cross-project contamination (each session checks its OWN PID)

Terminal Tabs vs. Windows

Important distinction: A "terminal window" can have multiple tabs. Each tab is an independent shell with an independent Claude process. Mission Control tracks sessions, not windows or tabs.

  • 1 terminal window with 3 tabs running claude = 3 sessions, 3 cards
  • 3 separate terminal windows, each with 1 tab = also 3 sessions, 3 cards
  • There is no API to determine which tab belongs to which window. This grouping is invisible to hooks.
  • Recommendation: Don't try to group by terminal window. Session is the right granularity. Each session = one conversation = one card. Keep it simple.

Edge Cases

User does /resume (new terminal) New session ID, new claude.exe PID. Old PID is dead. State file has stale PID. Hook detects dead stored PID via tasklist check, re-resolves to new PID. Card flips from "Done" to "Active" on first hook fire.

Claude crashes / OOM Process dies. No Stop hook. Same as tab close. Timestamp goes stale → auto-mark as done.
User runs claude in two different repos Two separate claude.exe processes, two session IDs. Hook writes go to each project's own states/ dir. Each Mission Control instance only reads its own states/ dir. No cross-contamination. (Unlike process counting which sees ALL claude.exe globally.)

Summary: What We Can and Cannot Track

  • CAN track: Which terminal windows are open (PID-based liveness)
  • CAN track: What state each agent is in (investigating, developing, idle, etc.)
  • CAN track: Every prompt the user types (append-only ndjson, combined across session chains)
  • CAN track: Every tool the agent uses (activity log)
  • CAN track: Context transitions within a terminal (/plan, /compact — via session chaining + ctx count)
  • CANNOT track: Which terminal window/tab a session belongs to (only the process, not the UI container)
  • CANNOT track: What happens during /compact or long responses (no hooks fire, but PID stays alive)

No timestamp thresholds needed. The PID check is binary: alive or dead. The only fallback is for legacy sessions that don't have a claudePid yet (they use a 120s timestamp threshold until a hook fires and captures the PID).

Identity Model
How Sessions Are Identified

Session ID Source

The hook reads data.session_id from stdin (provided by Claude Code). Fallback chain: process.env.CLAUDE_SESSION_ID → pid-{process.ppid}. The pid- fallback is unreliable on Windows because each hook invocation spawns a new child process with a different parent PID.

One Session = One Card

Session ID is a UUID generated by Claude Code at conversation start. It's stable across the lifetime of one conversation. /resume creates a new session ID but inherits the old conversation context. Sessions sharing a claudePid are merged into one card. If /resume happens in a new terminal, the hook detects the dead stored PID and re-resolves to the new process.

Terminal Tabs

Each terminal tab running claude gets its own session ID. Multiple tabs = multiple sessions. There is no "terminal window" grouping — each tab is independent. Closing a tab kills the process without firing the Stop hook.

What Claude Code Provides to Hooks

Via stdin JSON: session_id, tool_name, tool_input (for PostToolUse), prompt (for UserPromptSubmit). There is no "window ID", no "tab ID", no "close" event. The session_id is the only reliable anchor.

File Layout
What Lives on Disk

states/{sid}.json

Current snapshot. Overwritten on every tool call. Fields: sessionId, tool, detail, state, emoji, label, ts. The ts field is the key — it's the last time this session was heard from.

logs/{sid}.ndjson

Append-only activity log. One JSON line per state transition. Fields: state, emoji, tool, detail, ts. Deduped by state+tool to avoid flooding.

prompts/{sid}.ndjson

Append-only prompt log. One JSON line per user prompt. Fields: prompt, ts. Written by prompt-hook.js only. Server never modifies.

summaries.json

AI-generated one-liner summaries keyed by session ID. Contains summary, hash, ts per session. Generated by Cerebras on 30s loop. Hash is used to detect when prompts changed (stale summary).

Reference
Terminal Window Behavior

How Mission Control actually tracks terminals (verified):

  • Terminal identity = claudePid (the claude.exe process for that tab). Stable for terminal lifetime. Dies when tab closes.
  • Session ID is per-conversation, NOT per-terminal. Changes on /plan → implement, /compact, /resume.
  • One card = one PID. Multiple sessions sharing a PID are merged. Combined prompts, shared mission, context-switch count.
  • Liveness = is the claudePid in the live tasklist? If yes → active. If no → done. No timestamp threshold.
  • Process tree: claude.exe → bash → bash → bash → node hook.js. Hook walks 3 levels up via PowerShell Get-CimInstance to find claude.exe.
  • Subagents and background agents share the parent session ID (same card).
  • No close event exists on Windows. PID check replaces this entirely.

Implication: We CAN track terminal windows — via their claude.exe PID. One tab = one PID = one card. Session IDs are an internal detail that gets abstracted away by the merging layer.

What We Got Wrong

The original logic page was written by an AI agent analyzing the system from first principles. Several of its conclusions were wrong or incomplete. This tab documents the corrections discovered through actual testing on Windows Terminal, and the user's core requirements that the original analysis failed to account for.

Fundamental — User Requirements
The Goal Was Never "Timestamp Freshness"

What the original page said: "The only reliable signal that a session is still alive is the timestamp of its last hook event. If no event for >120s, the session is dead or the user has walked away."

Why that's wrong for the user's goal:

The user's mental model is: "If I have a terminal window open, that session is active." Not "if a hook fired recently." A user who opens 3 tabs, starts work in all 3, then sits in one thinking for 10 minutes — that session is alive. The terminal is open. The process is running. The agent is waiting. It should say active.

Timestamp-based liveness breaks this contract in multiple scenarios:

  • /compact — Claude Code compresses context internally. No hooks fire during this. Timestamp goes stale. Card drops to "done" while the agent is literally processing.
  • User thinking for 5+ minutes — Stop hook fired once at end of agent turn. No events after that. Timestamp-only approach marks this as "done" while the user is reading output and composing their next prompt.
  • Long agent responses — When Claude is generating a long response (not tool use), no PostToolUse hooks fire. The session appears dead to a timestamp check.
The correct goal: "Active" = the claude.exe process for that session is alive. Period. If the process exists, the window is open. If the process is dead, the window was closed. This is what the user wants, and it's what we can deliver.
Technical — Process Tree
ppid IS Walkable to claude.exe (The Logic Page Was Wrong)

What the original page said: "The hook's process.ppid is NOT the claude.exe PID — it's an intermediate process." … "Each hook invocation spawns a new child process. The ppid changes every time. Useless on Windows."

What's actually true:

Yes, process.ppid changes per invocation (it's a fresh bash.exe each time). But the original analysis stopped there and concluded process detection was impossible. It's not.

The actual process tree (verified via PowerShell Get-CimInstance Win32_Process):

claude.exe:70500          ← STABLE. Dies only when terminal closes.
  └─ bash.exe:28508         ← Intermediate (spawned by claude to run hook)
       └─ bash.exe:42444    ← Intermediate
            └─ bash.exe:49136  ← process.ppid for node hook.js
                 └─ node.exe:8064  ← The hook itself

Walking from process.ppid up 3 levels always reaches claude.exe. This was verified across multiple invocations — the intermediate bash PIDs change every time, but claude.exe PID is rock-stable for the session's lifetime.

How we walk it: A PowerShell script (find-claude-pid.ps1) uses Get-CimInstance Win32_Process to walk ParentProcessId until it finds a process named claude. Takes ~1.2 seconds. Runs once per session (first hook fire), result cached in state file forever after.

Correction: The truth table should mark process.ppid as: "Not directly useful, but walkable to claude.exe via PowerShell CIM query. Per-session PID detection IS possible."
Technical — Liveness Architecture
PID-Based Detection: The Correct Architecture

The old broken approach: Count ALL claude.exe processes globally, assign "active" to top N by recency. No per-session mapping. Cross-project contamination.

The timestamp-only approach (proposed by original logic page): If ts < 120s, active. Else done. Per-session, but fails during silence (compact, thinking, long responses).

The correct approach (PID-based):

// HOOK SIDE: Capture claude.exe PID once per session
if (!existingState.claudePid) {
  // Walk process tree: ppid → bash → bash → bash → claude.exe
  // via PowerShell Get-CimInstance (1.2s, one-time cost)
  claudePid = findClaudePid(process.ppid);
}
stateFile.claudePid = claudePid;

// SERVER SIDE: One tasklist call per poll (~130ms)
const livePids = parseTasklist(
  execSync('tasklist /FI "IMAGENAME eq claude.exe" /FO CSV /NH')
);

// Per-session check
for (const session of sessions) {
  if (session.claudePid) {
    session.active = livePids.has(session.claudePid);  // direct check
    session.done = !session.active;
  } else {
    // Fallback for sessions that haven't captured PID yet
    session.active = session.ago < 120_000;
    session.done = !session.active;
  }
}

Why this works for EVERY scenario:

ScenarioTimestamp-OnlyPID-Based
Agent making tool calls Active (ts fresh) Active (PID alive)
/compact running Drops to done (no hooks) Active (PID alive)
User thinking 10 min Drops to done Active (PID alive)
Long agent response (no tools) Drops to done Active (PID alive)
Tab closed Done (ts stale) Done (PID dead)
Crash / OOM Done (ts stale) Done (PID dead)
Sleep / wake Briefly "done" until hook fires Active (PID alive after wake)
Cross-project sessions No contamination No contamination

Performance:

  • Hook side: ~1.2s PowerShell call, runs ONCE per session (first hook fire only). Subsequent hooks: 0ms (PID already cached in state file).
  • Server side: Single tasklist call (~130ms) per poll covers ALL sessions. Well within the 2s poll budget.
Meta — Lessons
Why the Original Analysis Was Overconfident

The original logic page made definitive statements like "there is exactly one reliable way to detect session liveness" and "The only reliable signal is the timestamp." These conclusions were reached by theorizing about the Windows process model rather than testing it.

What was assumed vs. what testing revealed:

  • Assumed: "ppid is useless on Windows." Reality: ppid is not directly useful, but walking the tree to claude.exe works perfectly. The assumption stopped one step too early.
  • Assumed: No way to map a process to a session. Reality: The hook runs inside the claude.exe process tree. Walking up always finds the right claude.exe. Store the PID once, check forever.
  • Assumed: wmic available for process queries. Reality: wmic is deprecated/removed on Windows 11. Must use PowerShell Get-CimInstance.
  • Assumed: User's goal is "show recent activity." Reality: User's goal is "show which terminal windows are open." These are fundamentally different requirements.
Lesson: Always test process assumptions empirically on the target OS. And always ask the user what "active" means to them before designing the detection system.
