Demo Guide

Reference this before showing Mission Control to someone. What to show, what to say, what to skip, and how to handle questions.

1 Pre-Demo Checklist
Do these BEFORE sharing your screen.
  • MC server running: node .claude/agent-hub/server.js on port 3033
  • At least 1-2 Claude Code sessions active so Dashboard shows live cards with pulsing indicators
  • Dispatch has items — a mix of todo, in-progress, and done. Empty Dispatch kills the demo.
  • Findings page has content — currently 34 findings. Spot-check it loads.
  • Campaigns page shows Sprint 1 — debrief with wins/losses populated.
  • Browser zoom comfortable — MC has 1.3x zoom built in. Make sure text is readable on shared screen.
  • Home/Work toggle set to Home — shows the personal project context (photography portfolio + MC). More relatable than "work."
  • Close sensitive tabs — no email, Slack, or passwords visible in other browser tabs.

3 Page-by-Page: Impressive vs. Rough
Know what's strong and what to gloss over.
Dashboard/
Strong
  • Show: Live agent cards, pulsing indicators, tool/file display, session age
  • Show: Favicon changes color based on agent state (green = working)
  • Show: Click into a session card for detail view
  • Skip: Empty state if no agents running — always have agents active for the demo
Findings/findings
Strong
  • Show: Tier system (tier1 = critical principles, tier5 = pain points)
  • Show: Real user quotes — makes it visceral and authentic
  • Show: The sheer volume (34 findings) — proves the system is actively used
  • Emphasize: "These get loaded into every new agent session automatically"
Dispatch/dispatch
Solid
  • Show: Priority levels, status colors, task count
  • Show: Home/Work mode toggle (relatable for anyone with side projects)
  • Skip: Detailed task editing — it works but the UX isn't polished yet
Campaigns/campaigns
Solid
  • Show: Sprint 1 with 4 agents, wins/losses debrief
  • Emphasize: "Losses are just as valuable — they become findings"
  • Skip: The remaining items list can feel long — focus on wins/losses narrative
Toolbox/tools
Solid
  • Show: Agent roster, skill library, command catalog
  • Emphasize: "Composable — agents can use skills, skills can chain"
  • Skip: Individual agent details unless someone asks

WIP
  • Show briefly: Structure of a post-mortem — timeline, root cause, prevention
  • Skip: Don't dwell here unless they ask about the review process

Skip
  • These pages exist but aren't polished enough for a demo

4 Killer Moments to Hit
These are the lines and visuals that make people go "oh."
💡
"Without opening a single file"
When showing Dashboard: "I can see every agent, what they're working on, what they've learned, and what went wrong — without opening a single file." This is the hook. Land it early.
🔄
The learning loop in action
Pick a finding with a strong quote (f027: mispasted prompt is great). Read the quote out loud. Then: "This became a rule. Every future agent reads it. It can't happen again." Pause. Let it land.
🎯
"Losses become findings"
On Campaigns, show the losses. Then flip to Findings and find the corresponding entries. "See? The failure from last sprint is now a rule that prevents the same failure next sprint." The connection makes it real.
⚡
The simplicity numbers
"Zero npm dependencies. One server file. 34 findings captured. All local, all private." People don't expect this level of functionality from something so minimal.
🚀
"I ran 4 agents simultaneously"
On Campaigns: "I coordinated 4 agents in parallel, each with their own brief. One built the Campaigns page, one worked on terminal identity, one did strategic evaluation, one researched UX patterns." This sounds impressive because it IS impressive.
5 Handling Questions
Prepared answers for the questions people will actually ask.
"How is this different from just using Claude Code?"
Claude Code is the engine. MC is the cockpit. Claude Code has no visibility layer — you can't see across sessions, you can't capture institutional knowledge, you can't coordinate agents. MC adds the management layer that CLI doesn't have.
"Could this work with Cursor / Copilot / other AI tools?"
The architecture is hook-based — it observes events from the CLI. Right now it's built for Claude Code because that's what I use. But the concept (visibility + memory + coordination for AI agents) applies to any multi-agent workflow. The hooks would need adapting.
"Is this open source?"
Not yet, but heading that direction. It's a single server.js file with zero dependencies. The code is simple enough that anyone could read and modify it. Right now it's a personal tool that's becoming a product.
"How do the agents actually read the findings?"
Findings are stored as JSON. Key findings get promoted to CLAUDE.md (the project instructions file Claude Code reads on every session start) and to memory files that persist across conversations. Every new agent loads these automatically.
"How long did this take to build?"
MC is itself built with Claude Code agents. It's both the product and the proof of concept. The agents that build MC are managed by MC. That recursive loop is part of what makes it interesting — it's a self-improving system.
"Does this send any data externally?"
Nothing. Ever. Everything is local. Flat JSON files on disk. No analytics, no telemetry, no accounts, no cloud. Your data stays on your machine.
"What's the catch?"
It's early. Some pages are rough. The architecture is opinionated (flat files, no database, single server). But that's the point — it's simple enough to understand completely, and it evolves with every session. You need to be running Claude Code to get value from it.
